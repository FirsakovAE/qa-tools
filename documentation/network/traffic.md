# Breakpoints и Mock response

Во вкладке **Network** и в настройках **Options → Network** доступны два связанных, но разных механизма:

- **Breakpoint** — остановить запрос или ответ, когда URL совпал с правилом, **посмотреть или изменить** данные и только потом отпустить выполнение дальше (уже по настоящей сети, если вы не отменили изменения).
- **Mock response** (в коде и настройках часто рядом с идеей **Map Local**, как в Charles) — **не ходить в сеть** и сразу вернуть странице **синтетический ответ** с заданным кодом, заголовками и телом.

Базовые ограничения перехвата (только `fetch` / XHR, контекст страницы, CORS и т.д.) описаны в [Работа с запросами](/network/general).

## Сравнение с Charles Proxy

**Charles** (и аналогичные **системные / внешние прокси**: Fiddler, mitmproxy и др.) встаёт **между клиентом и сервером** на уровне ОС или отдельного приложения:

- видит трафик **не только из браузера**, но и из мобильных приложений, CLI, других программ (если направить их через прокси);
- для **HTTPS** обычно нужна установка доверенного сертификата и **SSL Proxying** — расшифровка TLS на прокси;
- **Breakpoint** и **Map Local / Remote** применяются ко **всем** подходящим запросам в этом канале.

**Vue Inspector** работает **внутри вкладки браузера**: подменяются глобальные **`fetch`** и **`XMLHttpRequest`**.

| Аспект | Charles (типично) | Vue Inspector |
| ------ | ----------------- | --------------- |
| Уровень | Прокси / вне процесса браузера | JavaScript на странице |
| Охват | Много клиентов, весь HTTP(S) через прокси | Только этот документ: вызовы **fetch** и **XHR** |
| HTTPS | Свой корень доверия, разбор TLS | Без отдельного «взлома» TLS — приложение и так отдаёт ответы в свой JS |
| Service Worker, нативные запросы | Можно перехватывать на пути | Не перехватываются этим механизмом |
| Удобство для SPA в браузере | Нужна настройка прокси и сертификатов | Включил правила в инспекторе — работает на текущем сайте |

Итог: по **смыслу** Breakpoint и Mock в Inspector близки к Charles (**остановить / подменить ответ**), но по **месту действия** это не сетевой прокси, а **инструмент разработчика веб-страницы** на том же origin-жизненном цикле, что и ваше Vue‑приложение.

## Контекст работы с веб-сайтом

Имеет смысл использовать эти функции, когда:

- нужно **отладить фронтенд** без стабильного бэкенда (mock ответа API);
- нужно **изменить один запрос** (заголовок, тело, URL) и посмотреть, как отреагирует UI;
- нужно **имитировать задержку** сети (у mock есть поле **delay** в миллисекундах — перед возвратом фиктивного ответа выдерживается пауза, по аналогии с **Throttle** / задержками в Charles).

Ограничения:

- запросы, которые **не идут через fetch/XHR** (например, обычная загрузка `<img src="…">` без вашего кода), **не попадут** под эти правила;
- **OPTIONS** (preflight) в перехватчике **исключены** — для CORS‑сценариев поведение может отличаться от полного прокси;
- если сработал **mock**, реальный HTTP‑запрос **не выполняется** — на сервере его не будет, в отличие от breakpoint, где запрос после «продолжить» обычно уходит в сеть.

**Порядок проверки при `fetch`:** сначала **mock** (если подошло правило — сеть и точка останова **на запросе** для этого вызова не используются), иначе обычный путь с **breakpoint** и реальным запросом. Имейте в виду при совпадающих URL: активный mock «перекрывает» сценарий с остановом до сети.

## Breakpoints (точки останова)

### Назначение

Правило задаёт **шаблон URL** (схема, хост, порт, путь, строка запроса), опционально **метод** и **когда останавливаться**:

- только до отправки (**request**);
- только после ответа, до того как страница его обработает (**response**);
- или **оба** варианта по очереди (**both**), если для запроса сначала срабатывает останов на запросе, а после ответа снова проверяется правило на **response** — см. фактическое поведение в интерфейсе при настройке **trigger**.

При совпадении выполнение **реально приостанавливается** (промис в перехватчике ждёт, пока вы в панели не **подтвердите** изменения или не **отмените** останов).

### Где настраивать

- Раздел **Options → Network → Breakpoints** — список правил, включение / выключение, редактирование, удаление.
- Вкладка **Network** — создание и привязка к конкретному перехваченному запросу (формы **breakpoint** из контекста записи).

Синхронизация активных правил идёт во **внедрённый скрипт** страницы; без загрузки инспектора правила не применяются.

### Что можно менять

- **На запросе:** метод, компоненты URL (схема, хост, путь), query‑параметры, заголовки, тело.
- **На ответе:** код, текст ответа (если применимо), заголовки ответа, тело ответа.

Список записей в Network может показывать «ожидающие» breakpoint; после **Apply** запрос продолжается с изменениями или ответ подменяется и возвращается приложению.

### Пауза записи (Pause) и breakpoints

Режим **Pause** отключает **добавление новых строк** в журнал, но **точки останова** по правилам могут по-прежнему срабатывать (в том числе уведомление UI на **response** breakpoint). Для повседневной работы удобнее не путать «не пишем в лог» с «не перехватываем».

## Mock response (Map Local)

### Назначение

Для совпавшего с правилом **метода** и **URL** вызова `fetch` или `XHR` перехватчик **не вызывает** исходный сетевой стек: собирается объект ответа **Response** / эмуляция **XHR** с полями из правила:

- HTTP‑**status** и **statusText**;
- массив **headers**;
- **body** (часто JSON‑строка);
- опционально **delay** перед возвратом.

Запись всё равно может появиться в списке **Network** как «мокнутый» запрос/ответ (для отладки и истории).

### Шаблоны URL

Для **хоста** используется сопоставление с поддержкой символа **`*`** (подстановка подстроки). **Путь** и **query** задаются с шаблоном, где `*` разворачивается в произвольный фрагмент (как префикс пути в регулярном выражении). Точное поведение при конфликте нескольких правил: срабатывает **первое подходящее** правило в порядке перебора активных моков в инжектируемом коде.

### Где настраивать

- **Options → Network → Mocks Responses**.
- Вкладка **Network** — действие вроде **Mock Response** у выбранного запроса (подсказка в настройках: создавать мок из журнала).

### Пример правила (иллюстрация структуры)

Ниже не готовый импорт, а ориентир полей, которые вы задаёте в форме:

```json
{
  "enabled": true,
  "method": "GET",
  "scheme": "https",
  "host": "api.example.com",
  "path": "/v1/user*",
  "status": 200,
  "statusText": "OK",
  "delay": 300,
  "headers": [
    { "name": "Content-Type", "value": "application/json" }
  ],
  "body": "{ \"id\": 1, \"name\": \"test\" }"
}
```

Для кодов **204**, **205** и **304** тело ответа не задаётся (по смыслу HTTP). Если JSON в **body** некорректен при заявленном `Content-Type: application/json`, строка может быть дополнительно нормализована предупреждением в консоли.
